# 考勤排班数据分析全流程

## 概述

将考勤原始Excel数据经过清洗、指标计算、透视分析和报告生成，产出完整的考勤分析结果。整个流程经过多轮验证和修正，确保业务规则100%准确。

## 数据流

```

                 │
```

## 执行流程

### 步骤1: 数据清洗

**脚本**: `scripts/clean_data.py`

读取7个原始文件，按严格顺序执行7步清洗：
1. 剔除离职人员（最后工作日 < 标准日期 且 审批状态含目标值）
2. 剔除未入职人员（入职日期 > 标准日期）
3. GL00工号处理（剔除GL502563 + 仅保留7个白名单工号）
4. 剔除 EU人力资源部
4.5. **剔除GUS白名单人员**（读取两个Sheet的工号并去重后剔除，通常为年薪制管理人员）
5. 补签数匹配：读取补签管理.xlsx → 过滤审批状态(已完成/审批中/转交) → 过滤非标准日期 → 按工号计数合并
6. 累计工时匹配：读取三份签字报表 → 按工号汇总每日总工时 → 本周/上周(减40, floor 0) + 双周(直接累加) → 合并

**执行前**:
- 确认 `OUTPUT_DIR` 配置正确
- 确认花名册文件名中的序号正确（如 `花名册 (12).xlsx`）
- 所有文件以 `dtype=str` 读取

### 步骤2: 指标计算

**脚本**: `scripts/add_metrics.py`

读取清洗后数据 + 班次字典，按规则0~5计算指标列。**规则执行顺序严格固定**：

- **规则0**: 休息开始/结束时间（与班次字典匹配，插入在 `班次名称` 列后）
- **规则1**: HUB标记（部门含`.H`或等于`EWR.G`/`CNO.G`，插入在 `五级部门` 列后）
- **规则2**: 是否排班正确（7级优先级判断，插入在 `末打卡时间` 列后）
- **规则3**: 每日总工时计算（原公式+居家办公审批中，插入在 `居家办公合计（审批中）` 列后）
- **规则4**: 是否日超8H（插入在 `日超8H` 列后）
- **规则5**: 是否排班 + 标准打卡数 + 缺卡数（插入在 `班次内打卡次数` 列后）
  - 缺卡数 = max(0, 标准打卡数 - 班次内打卡次数)

### 步骤3: 透视分析

**脚本**: `scripts/pivot_analysis.py`

按三级/四级/五级部门分组汇总，产出5个Sheet的Excel：

| Sheet | 分组维度 | 核心指标 |
|-------|---------|---------|
| 日超8H | 三级+四级 | 超过小时合计 |
| 是否排班 | 三级+四级 | 已排班数、排班率 |
| 是否排班正确 | 三级+四级 | 正确数量、正确率 |
| 打卡率 | 三级+四级 | 打卡数、打卡率、缺卡数、补签率 |
| hub排班是否正确 | 三级+四级+五级 | HUB正确率（仅hub人员） |

**分组规则**: 四级非空按四级分行 → 四级为空汇总为"/" → 每个三级末尾强制汇总行

### 步骤4: 交互式 HTML 报告（含穿透明细）

**步骤 4a — 构建报告数据**

**脚本**: `scripts/build_report_data.py`

从 `指标计算后数据.xlsx` 提取完整 JSON 数据，包含三个层级（全公司 / 三级部门 / 四级部门）的汇总统计 + 每个指标的穿透工号明细列表。

JSON 结构:
- `meta`: 元数据（总人数、日期范围、部门数）
- `overview`: 全公司汇总（所有指标 + 穿透员工列表）
- `departments[]`: 按三级部门分组
  - `summary`: 三级汇总（各指标计数 + 穿透员工列表）
  - `sub_depts[]`: 四级明细（每个四级的汇总 + 穿透员工列表）

**步骤 4b — 生成交互式 HTML**

**脚本**: `scripts/render_report.py`

基于 `report_data.json` 生成自包含的交互式 HTML 报告页面。

### 步骤5: 同比变化注入（v11新增）

**步骤 5a — 计算对比数据**

**脚本**: `scripts/compute_comparison.py`

读取昨天(前次)的 `透视分析.xlsx` 与今天的 `report_data.json`，逐大区逐指标对比，生成 `comparison_data.json`。

**用法**:
```bash
python compute_comparison.py <今天工作目录> <昨天工作目录>
```

**步骤 5b — 注入HTML页面**

**脚本**: `scripts/inject_comparison.py`

读取 `comparison_data.json`，将同比变化标识(CSS样式+JS逻辑)注入到所有8个HTML页面。

**用法**:
```bash
python inject_comparison.py <工作目录>
```

**注入效果**:
- 每个KPI卡片数值旁显示箭头 + 百分比变化（如 `↑ +47.2%`）
- 颜色规则:

| 指标 | 上升箭头颜色 | 下降箭头颜色 | 原因 |
|------|:---:|:---:|------|
| 超标率 | 🔴红 ↑ | 🟢绿 ↓ | 越低越好 |
| 排班率、正确率、打卡率、HUB正确率 | 🟢绿 ↑ | 🔴红 ↓ | 越高越好 |

- 持平标记: `→ 持平`（灰色）
- 通过CSS类 `.cmp-badge` + MutationObserver 自动在页面渲染后附加到卡片上

### 步骤6: SSO 权限守卫注入（v21新增）

**脚本**: `scripts/auth_injector.py`

在所有生成的 HTML 报告中注入飞书 SSO 权限守卫脚本，实现：

- **未登录拦截**: 无 JWT token → 自动跳转 `login.html`
- **区域权限控制**: 大区页面（如 `region_WE.html`）只允许对应区域 BP 访问；Admin 可访问全部
- **后端离线保护**: 后端不可达时不阻断页面，只输出警告

**用法**:
```bash
python auth_injector.py <HTML目录> [--backend-url http://...] [--login-page login.html]
```

**注入原理**:
- 在 `</head>` 前插入 `<script>` 块
- 脚本加载时立即校验 JWT（隐藏页面 → fetch `/api/auth/me` → 验证通过后显示）
- 区域页面设置 `PAGE_REGION` 变量，与 `user.pages` 做交集判断
- 注入后的页面可独立部署到 GitHub Pages

**配套文件**:
- `scripts/auth-guard.js` — 权限守卫逻辑（非压缩版，供调试参考）
- `scripts/login.html` — 飞书 SSO 登录页（深暗玻璃态，与报告风格一致）

**配置**:
部署时修改 `SSO_BACKEND_URL` 变量（HTML 内嵌），指向实际后端地址。

**交互功能**:
- 🏠 **全览仪表盘**: 首页展示全局指标卡片 + 部门总览表 + **汇总柱状图（前12部门KPI对比，位于表格下方）**

## 关键业务规则

**是否排班正确的7级优先级**（参见 `references/business_rules.md` 完整说明）:

```
1. 班次名称为空 → "/"（最高优先级，直接返回）
2. 休息日/节假日 + 首末打卡均为空 → "正确"
3. 班次不为空 + 首打卡为空 → |末打卡-班次下班| ≤ 1h ? "正确" : "不正确"
4. 班次不为空 + 末打卡为空 → |首打卡-班次上班| ≤ 1h ? "正确" : "不正确"
5. 休息日/节假日 + 单边打卡(首空末不空/首不空末空) → "不正确"
6. 其他（含非休息日双非空、非休息日双空等）→ "不正确"
```

## 使用方式

### 完整流程执行（v12: 单一入口）

```bash
# 方式1: 完整流程（推荐）
```

## 已知陷阱

1. **原始数据第一行可能是重复列头**: 检查 `考勤日期` 列第一行值，若重复则跳过
2. **花名册文件名含序号**: 如 `花名册 (12).xlsx`，不同批次序号不同，执行前需确认
3. **每日总工时列名有3种变体**: 脚本已按优先级尝试匹配
4. **班次时间字段可能带日期前缀**: 如 `1900-01-01 00:30:00`，parse_hm 已处理
5. **`/` vs `正确`**: 班次为空返回 `/`（不参与判断）；仅休息日/节假日且首末均为空返回 `正确`；非休息日双空返回 `不正确`
6. **标准打卡数与排班正确性的联动**: 备注(GF)时间段的判断逻辑取决于 `是否排班正确` 的值
7. **report_data.json 可能较大**: 包含所有穿透明细时可达数MB，HTML 内嵌后文件也相应变大，属正常
